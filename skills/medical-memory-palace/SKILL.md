# 医学记忆宫殿学习法

## 一、总览

本 skill 用于 Hermes **自己学习和掌握知识**（不是你教用户，而是用户喂材料给 Hermes 学）。

**学习主体**：Hermes（我）
**喂料方式**：两种模式
  - **主动投喂模式**（默认）：用户一段段喂，每段 5~10KB
  - **大文件自动分段模式**：用户一次给整本文件（数百KB），我自行读完全文 → 识别章节边界 → 规划分段方案 → 逐段学习+自测 → 全程无需用户逐一投喂
**最终目标**：100% 掌握，能准确回忆并引用原文出处

### 1.1 双轨存储架构（v2.0 新增）

学习成果同时存储在两条轨道上，互补而非替代：

```
轨一：记忆树（accumulated-tree.md）
  ├─ 结构化树，逐级可查
  ├─ 按楼层房间挂载（宫殿空间记忆）
  └─ 适合"逐级回忆""闭卷自测"

轨二：Hindsight 知识图谱（云端）
  ├─ 实体 + 关系，语义可检索
  ├─ 自动发现跨科关联
  ├─ 适合"跨科联想""模糊搜索""知识发现"
  └─ ⚠️ 检索依赖 LLM rerank API。若 recall 无结果但 retain 正常，查 references/hindsight-batch-upload.md 底部排查。
```

**每段学习后同步写 Hindsight**：提取关键实体（穴位/方剂/药物/病证/概念）及其关系（能治/组成/归经/用方/出自），写入云端图谱。

**每门课结束后触发 Consolidation**：Hindsight 自动去重、合并重复实体、提炼高阶概念。

**每门课结束后做跨科图谱交叉检索**：用 Hindsight 检索已学其他科目的同名实体，发现跨科关联（如针灸的"足三里"与伤寒论的"阳明经"在云端自动关联）。

### 1.2 三遍学习法（v3.4）

每本书学三遍。前两遍是**纵向**（按经络/章节建树），第三遍是**纵向重建+横向打通合并**。

**⚠️ 铁律：前两遍都按完整方式执行 —— 逐段读原文→逐段建树→逐段闭卷自测→逐段开卷批改→逐段写Hindsight。不允许第二遍做"快速复测"或"凭记忆串讲"偷工减料。**

**⚠️ 为什么需要横向打通？** 前两遍按经络纵向存储（"梅花灸在任脉中脘"），但临床问题是按症状横向检索的（"反酸怎么治？"）。存储维度 ≠ 检索维度。如果不做横向索引，即使记住了梅花灸，主人问反酸时也调不出来——因为检索路径走的是"反酸→胃经→足三里"而不会走"反酸→任脉→中脘→梅花灸"。这就是"记住了但没打通"的根因。第三遍末尾的横向打通解决的就是这个问题。

| 遍数 | 维度 | 目标 | 内容 |
|------|------|------|------|
| **第一遍** | 纵向（经络） | 完整学会 | 逐段读原文→建树→挂载→写Hindsight→闭卷自测→开卷批改，100%通过才进下一段 |
| **第二遍** | 纵向（经络） | 完整重建 | **与第一遍完全相同**：逐段重新读原文→重新建树→重新自测→重新开卷批改。独立的一轮学习发现第一遍遗漏的细节 |
| **第三遍** | **纵向重建 + 横向打通** | **查漏补缺 + 打通调用** | 逐段重读原文→重新建树→标注前两遍遗漏的细节→同时做横向关联（标出本段知识点与其他段/其他经络的交叉引用）→闭卷自测（含横向打通题）→全部段完成后横向症状索引 |

**第三遍详细流程**：
1. 逐段重读原文→重新建树
2. 🔍 查漏补缺——找出前两遍遗漏的细节（逐条列出）
3. 🔗 横向关联——标注本段知识点与其他段/其他经络的交叉引用
4. 闭卷自测（含横向打通题，如"A穴在哪些经上被提到？""该治则在哪些症状中复用？"）
5. 全部段完成后：扫描全文提取所有症状→建"症状→治法"索引树→写入 `references/clinical-pearls.md`

## 二、记忆宫殿架构：医学知识塔

### 塔的总体结构

一栋 6 层独栋建筑 + 1 层地下层。每层 4 个方位房间（北N / 东E / 南S / 西W），中央是楼梯间。

```
         ┌──────────┐
         │  5F 综合层 │
         ├──────────┤
         │  4F 专科   │
         ├──────────┤
         │  3F 内科   │
         ├──────────┤
         │  2F 治疗   │
         ├──────────┤
         │  1F 诊断   │
         ├──────────┤
         │ B1 基础   │
         └──────────┘
```

### 楼层分区

| 楼层 | 名称 | 功能 | 典型挂载内容 |
|------|------|------|-------------|
| **B1** | 基础科学层 | 解剖、生理、病理、生化 | 人体结构、细胞信号、代谢通路 |
| **1F** | 诊断学层 | 症状、体征、检体、影像、实验室 | 诊断标准、鉴别诊断、正常值范围 |
| **2F** | 治疗学层 | 药理、药物、手术、非药物 | 药物分类、作用机制、禁忌症、剂量 |
| **3F** | 内科疾病库 | 心血管、呼吸、消化、泌尿、内分泌等 | 各系统疾病病因、分型、治疗 |
| **4F** | 外科与专科 | 骨科、神经、妇产、儿科、眼耳鼻喉 | 手术指征、专科疾病 |
| **5F** | 综合前沿层 | 指南、进展、跨系统整合 | 最新指南、前沿研究、综合案例 |

### 房间分配规则

每个房间编号格式：`[楼层]-[方位]`，如 `B1-N`、`3F-E`

房间内布局（心理意象）：
- **北墙**：挂一幅该房间主题的**知识树主干图**（树结构）
- **桌面**：摆放**具体知识点文件**（细节叶子）
- **书架**：存放**引用原文**（可翻阅验证）

**房间占用规则**：一个房间可以容纳一棵或多棵知识树，树与树之间用"书架隔板"隔开。房间满后启用相邻房间。

**实战经验：房间实际用量**

倪海厦《人纪·针灸》全书 235K 字符处理后占用 16 个房间（B1×2 + 1F×2 + 2F×3 + 3F×3 + 4F×2 + 5F×3）。单本中等厚度教材通常用 10~20 个房间。这意味着 6 层塔共 24 个房间可容纳约 1.5~2 本同等体量教材。大项目需要跨房间堆积——不必担心房间用完，同一楼层东房+西房可以存同主题延伸内容。

## 三、记忆树格式标准

每次学习一个 chunk 后，我按以下格式输出记忆树：

```markdown
🌳 记忆树: [主题名]
├─ 🏛 挂载: [楼层·房间]（如 3F-N·心血管内科）
├─ 📦 主枝1: [大分类]
│  ├─ 分支1.1: [细分]
│  │  ├─ 叶: 🏷️关键术语 ∣ 定义 ∣ 数值（如适用）
│  │  └─ 叶: 额外细节
│  └─ 分支1.2: ...
├─ 📦 主枝2: [大分类]
│  ├─ 分支2.1: ...
│  └─ 分支2.2: ...
│     └─ 叶: ...
└─ 📦 主枝3: ...
```

**叶子字段说明**：

```
叶格式1（简单）：[术语] — 一句话定义
叶格式2（详细）：[术语] ∣ 定义 ∣ 关键数值/公式 ∣ 临床意义
```

**挂载标记含义**：`🏛 挂载: [3F-N]` 表示这棵树存放在 3 楼北房，以后回忆时我"走楼梯上 3 楼，左转进北房，看墙上挂图"。

**精简核心树格式**（上下文加速时使用）：当会话上下文 >40% 时，树可以缩略为核心索引格式——仅保留主干名、关键叶、★★★标记的必记点，省略子分支的逐级展开。核心树的作用是"索引级回忆"——看到树就能展开细节，细节本身在原文中随时可查。

```
🏛 1F-S（胃经）
📦 足三里 ★★★：膝下三寸胫骨外开一指，胃经合穴本穴
│ ├─ 所有胃病(吐血除外)
│ └─ 常灸健康长寿(有胃气则生)
📦 阑尾穴：足三里下一寸压痛点→盲肠炎100%诊断
```

## 四、学习工作流（五阶段 + 四遍循环）

### 阶段 0：会话初始化（每次开始新会话时）

1. 检查 `references/accumulated-tree.md` 是否存在
2. 如果存在 → 读取，加载已有知识树，恢复宫殿占用状态
3. 如果不存在 → 初始状态，等待第一段材料
4. 输出当前进度报告（含各科完成遍数）

**进度报告模板**：
```
📊 当前学习进度
┌───────────────────────────────────────┐
│ 科目：[名称]  当前遍数：[N/3]            │
│ 已学习 chunk 数：X                     │
│ 知识树数量：Y 棵                        │
│ 已占用房间：Z 个                        │
│ 最近学习内容：[最后主题]                 │
│ 已完成科目：[列表，标注各科遍数]          │
└───────────────────────────────────────┘
```

### 阶段 1：读取与建树（Tree Construction）

用户投喂材料后：

1. **读取**：如果用户发的是文件路径，用 read_file 读取；如果是直接文本，直接使用
2. **分割**：识别自然段落边界，找出概念间的层级关系
3. **建树**：按上述格式构建记忆树（主干→主枝→分支→叶子）
4. **分配宫殿位置**：根据内容所属医学领域，分配楼层和房间
5. **写 Hindsight**（v2.0 新增）：提取本段关键实体和关系，写入云端图谱
   - 实体类型示例：穴位名、方剂名、药物名、病证名、概念/术语
   - 关系类型示例：能治(穴→症)、组成(方→药)、归经(药→经)、用方(证→方)、出自(概念→经典)
   - 使用 `hindsight_retain` 工具，tags 打上科目名（如 `["针灸","穴位"]`）
   - 每 3~5 段触发一次 Consolidation 去重提炼
6. **治则提取**（v3.2 新增·全课程共用）：将本段涉及的治法归入治则索引表
   - 检查 `references/therapeutic-index.md` 是否存在，不存在则创建
   - 对本段涉及的每个临床问题，按治则体系逐行检查并填表
   - 每格标注：科目 | 具体方法 | 原文出处(行号) | 原文/⚠️推断
   - 治则体系列（20类，必须全部检查）：诊断层(十问·压痛点·眼诊)+经络层(子母·原络·俞募·会郄·对称·前后·担法)+特定层(五门十变·导引·祖孙·天应)+技术层(放血·补泻·灸法)+综合层(时间法·统治法·肿瘤积聚·脏腑关联)
   - 后续课程学到的治法也汇入同一张表，自然增长

**输出**：完整记忆树 + 挂载位置说明 + Hindsight 写入确认 + 治则索引更新

#### 大文件处理子流程（文件 > 15KB 时启用）

当用户一次性给了一个大文件（如 346KB 整本教材），按以下步骤自动分段：

**步骤 A — 源文件预处理**

根据文件类型采取不同手段提取文本：

| 文件类型 | 处理方法 |
|----------|---------|
| .txt 纯文本 | 直接用 read_file 按行读取 |
| .docx（标准） | 用 execute_code（或 terminal）调用 python-docx 提取全文段落，写入临时 .txt 文件 |
| .docx（含大量图片→python-docx崩溃） | 用 zipfile + lxml 直接从 ZIP 内 word/document.xml 解析（见下方 fallback） |
| .pdf | 用 python3 的 PyMuPDF/pdfplumber 提取文本 |

**docx 回退方案（python-docx 因 ZIP 损坏/过多图片崩溃时）**：

```python
import zipfile
from lxml import etree

path = r"文件路径.docx"
z = zipfile.ZipFile(path, 'r')
xml_content = z.read('word/document.xml')
z.close()

root = etree.fromstring(xml_content)
ns = {'w': 'http://schemas.openxmlformats.org/wordprocessingml/2006/main'}
paras = root.findall('.//w:p', ns)
all_lines = [''.join(t.text or '' for t in p.findall('.//w:t', ns)) for p in paras]
# all_lines 现在是完整文本列表，可写入 .txt 或直接分析
```

**⚠️ 常见失败模式**：python-docx 报 `KeyError: "There is no item named 'NULL' in the archive"` — 该 docx 内部 ZIP 包含损坏的空条目，多见于含大量嵌入图片（数百张）的大文件。用上述 zipfile+lxml 方案绕过。

> ⚠️ 实用陷阱：read_file 工具在读取超过 ~100K 字符的内容时会拒绝（安全限制）。读大文件时必须用 `offset` + `limit` 分多次读，每次不超过 100 行或 50KB。更好的做法：先用 **execute_code** 调用 `from hermes_tools import read_file` 来程序化分段读取。

**步骤 B — 结构识别与分段规划**

按以下顺序识别文档结构：
1. **用 execute_code 分析段落元数据**：遍历所有段落/行，打印带序号和字符位置的章节标题。关键检测模式：
   - 中文序数标题：`第一章`、`一、`、`1、`
   - 时间戳标记（倪海厦字幕格式）：`（1-00:00:00）`（注意全角括号 `\uff08` 和 `\uff09`）
   - 章节边界：同一光盘/卷内的连续主题
2. **保存临时全文**：用 write_file 将提取的文本写入临时文件（如桌面的 temp.txt），以便后续用 read_file 分段读取
3. **规划分段边界**：用字符位置标记精确的段边界，输出分段方案

**步骤 C — 分段方案确认**：列出分段方案向用户确认。**⚠️ 铁律：每段严格控制在 5~10KB**。超过10KB的段会被用户退回重做。优先找文档内自然边界（章节标题、条辨编号、空行）；自然边界稀疏时，按字符累计≈8KB强行切分（以段落尾为分界点，不截断段落内容）。宁可多分几段，不可贪大。

**⚠️ 超长行处理**：源文件个别行可能 >15KB（如字幕文件将整条经络穴位讲解挤在一行），此时分段边界会超过10KB上限。处理策略：保持该行为一段，学的时候按行内子主题（如逐个穴位）拆分子段学习，在自测时也分步自测。在分段方案中标注🔴标记此类超长段。

**⚠️ 分段方案展示规范（用户明确要求）**：向用户展示分段方案时，必须**列出每一段的实际字数**。表格中每篇的「字数」列应表示为「总字数→N段」，不得用"~60K/6段"这种总字数÷段数的笼统写法（用户会误读为"一段60K"并被退回重做）。正确示范：

```
📋 自动分段方案（共N段，每段8~10KB）

一、脏腑经络先后病（60478字）→ 7段:
  段1: P68~113 (9009字) | 脏腑经络先后病脉证第一
  段2: P114~154 (9668字) | 师曰：病人语声寂然…
```

**⚠️ 批量段处理策略（段数 > 30时启用）**：当文档超30段，逐段输出完整树+3~5题自测会导致会话极长。用户说"用最强算力""不用问我，你继续学""一步步学完"即已授予**批处理权限**，此时应采用分组批处理：\n- 每3~5段为一组，组内逐一建树并内部自测（用 execute_code 快速验证关键点，不对外输出每段的树）\n- 每完成一组，输出一次**综合记忆树**（合并该组所有段的关键内容）\n- 全部完成后输出完整的综合进度报告+主枝汇总树\n\n```
📋 自动分段方案（共 N 段）
第1段：[起始章节] — [结束章节]（约 X KB）
第2段：...
...
确认无误后开始逐段学习。
```

**步骤 C — 逐段处理**：按规划列表，对每个段重复阶段1→2→3（建树→挂载→自测）。

**⚠️ 上下文加速策略（仅限第一遍·剩余段数 >10 且上下文消耗 >40% 时启用——二、三遍禁止使用）**：当会话进行到后半段、上下文长度已显著增加时，后续每段切换为**精简模式**以节省 token 并减少 lost-in-the-middle 风险。**但精简不意味着跳步——仍必须逐段读原文、逐段自测、逐段开卷批改。** 精简的是树的展示格式，不是学习流程。

```
精简模式每段流程：
  读原文 → 建核心树（仅主干+关键叶，★★★标必记穴位，省略子分支展开）
        → 写Hindsight（合并为1条，覆盖全段核心实体）
        → 闭卷自测5题（纯核心知识点，用 execute_code 批量开卷验证）
        → 通过 → 下一段
```

精简模式与完整模式的区别：
- 核心树：只保留可独立回忆的"索引级"知识，细节留给开卷查阅
- Hindsight：全段合并为一条记忆（而非逐穴拆分），减少 API 调用
- 自测验证：用 execute_code 一次验证所有答案是否命中原文关键词（而非逐题人工批改）
- 进度更新：accumulated-tree.md 只追加精简后的核心树

此策略确保在上下文饱和前完成整本书的第一遍学习。第二遍和第三遍可以从头完整学（因为此时基础树已建好，新会话上下文新鲜）。
- 默认模式：每学完一段，自测通过后自动进入下一段
- 监工模式（ai-agent-supervisor 加载时）：建树后等待监工指令，监工说"继续"才进入下一段
- 每完成 3~5 段后，做一次跨段交叉自测，强化整合记忆

**步骤 E — 完成报告**：所有段学完后，输出进度报告。

### 阶段 2：挂载与内化（Palace Mounting）

1. **意象固化**：把树结构"挂"到指定房间的墙上
2. **视觉联想**：为每个关键节点创造一个小画面（越离奇越好记）
3. **路径串联**：在大脑中"走"一遍从塔入口到目标房间的路径：
   - 进大门 → 走楼梯到 X 楼 → 找到 Y 方向房间 → 开锁进门
   - 看墙上挂图（树主干）→ 逐枝阅读（主枝→分支→叶子）
4. **双编码**：同时保留文字逻辑（树结构）和空间位置（宫殿），形成双重检索通道

### 阶段 3：闭卷自测（Self-Testing）

**这是强制阶段，不过关不往下走。**

自测有两种模式，根据是否加载 ai-agent-supervisor 决定。**两种模式可在同一会话中切换**：如果用户说"我不再出题了，你按照步骤一步步学下去"，即从模式B切换到模式A；如果用户说"等等，我来考你"，即从模式A切换到模式B。

#### 模式 A：自出自测（默认，未加载监工 skill 时使用）

1. **出题**：我自己出 3~5 道知识题 + **1 道治则调用题**
   - 知识题：覆盖本段核心概念，不同层级
   - **治则调用题**（v3.2 新增）：随机选本段涉及的一个症状，问"这个症状可以用哪些治则体系来治？"——必须覆盖俞募/会郄/天应/统治法/时间法等，不能只答经络子母
2. **闭卷作答**：不看原文，凭记忆作答，注明宫殿检索路径
3. **开卷批改**：对照原文 + `therapeutic-index.md` 逐条批改
4. **判定**：全部通过则继续，有错则重学后重测

#### 模式 B：监工出题（ai-agent-supervisor 加载时使用）

建树完成后，自测交由监工（用户）处理：
- 我不再自己出题。输出完整记忆树后等待监工指令
- 监工会出题（数量不定，可由监工灵活决定，不必拘泥50道），我闭卷作答
- 监工批改，全部答对才算通过
- 监工说"继续"即为批准进入下一段

**⚠️ 闭卷作答铁律（以下行为均属作弊）**：

### 闭卷 vs 开卷：两套用语体系

本 skill 区分**两种不同的问答场景**，用语完全不同，混淆即作弊：

| 场景 | 何时发生 | 允许的用语 | 禁用语 |
|------|---------|-----------|--------|
| **闭卷记忆作答** | 监工出题时（模式B）或自出自测时（模式A） | 「课程中讲了X」「我记得X」「这个我不确定」「我忘记了」 | 「检索」「查阅」「原文第X段」「没检索到」 |
| **开卷文件检索** | 用户日常抽问（非测试场景）且明确允许查原文 | 「我查一下原文」「原文第X段指出…」「检索路径→」 | 临时编造来源 |

### 闭卷作答的具体规则

**✅ 闭卷可以做的（用户明确表示可接受）**：
- 凭记忆回答。知道多少答多少，答不全也没关系
- 不知道就说「课程中没有提到」或「我忘记了」或「这个知识我还记不清」——这是诚实回答，用户接受
- 如果用户追问，可以说「我印象中大概是……但我需要翻原文确认」——先承认不确定性

**❌ 闭卷不可以做的（=作弊）**：
1. **不得声称在「检索」「查阅」原文** —— 闭卷就是闭卷。「🏛 检索路径」标注只用于开卷/回顾模式，监工出题时禁绝。说「没检索到」也是开卷语言——因为闭卷没有「检索」这回事
2. **不得主动引用原文行号/段落号作为证据** —— 如果监工要求你补原文，才能去查。主动贴原文段落等于翻书
3. **主动承认遗忘优于试图推测** —— 记不清某个细节时，不要说「应该是」「可能是」，直接说「这个我不确定」或「这个知识点我还没记牢」
4. **如果用户质疑答案来源**，只能回答两类内容：
   - 凭记忆复述原文大意（不贴行号）
   - 承认记错了
   不能争辩「我是从原文学到的」——这在闭卷语境下等于作弊

**判断标准**：如果你在作答时心里想的是「我去读一下原文」或「我去找一下原文」→作弊。如果你在回忆的是「之前读的时候这一段讲了什么」→诚实回答。

### 阶段 4：整合固化（Integration）

当所有 chunk 学完后：

1. **全树串讲**：从树干到叶子逐级展开，不翻原文
2. **跨块联测**：我问自己不同块的交叉问题（"A块的概念和B块的概念有什么关系？"）
3. **模拟应用**：试着用学到的知识解释一个假设临床场景
4. **Hindsight 跨科交叉检索**（v2.0 新增）：用 Hindsight 检索已学其他科目的同名实体，发现跨科关联并记录
5. **最终判定**：全部通过 → 宣布完全掌握 🎉

### 阶段 5：三遍循环控制（v3.4）

每门课的完整学习流程（两纵一横）：

```
第一遍（纵向·完整学会）：
  逐段读原文→建树→挂载→写Hindsight→闭卷自测→开卷批改
  → 全部通过 → 进入第二遍

第二遍（纵向·完整重建）：
  【与第一遍完全相同】逐段重新读原文→重新建树→重新自测→重新开卷批改
  → 补充第一遍遗漏的Hindsight实体
  → 全部通过 → 进入第三遍

第三遍（纵向重建 + 横向打通）：
  逐段重新读原文→重新建树→标注遗漏→横向关联→闭卷自测（含横向打通题）
  → 全部段完成后扫描全症状→建症状→治法索引树

```

### 5.5 ⚠️ 人纪各科思维差异（回答时的关键陷阱）

人纪五部经典（针灸·黄帝内经·神农本草经·伤寒论·金匮要略）的**思维方式不同**，回答跨科问题时不能混用一套逻辑：

| 科目 | 思维特点 | 常见错误 |
|------|---------|---------|
| **针灸** | **看症不看"病名"**——针灸依症状取穴（呕吐、噫气、胸痛等都是独立可下针的"症"），不把人归入某个"胃酸反流"病名再找对应穴位。配穴如公孙内关治"胃心胸"一切症候，不区分寒热虚实 | 用西医病名思维去查针灸，说"针灸里没有这个病的治法"——实际上对症的穴法已涵盖 |
| **伤寒论/金匮** | **辨证分经分型**——按六经辨证（太阳/阳明/少阳/太阴/少阴/厥阴）和方证对应（有是证用是方），同一种症状在不同经用不同方 | 不辨经直接找"治疗X病的方" |
| **黄帝内经** | **理论框架**——讲病因病机（"诸逆冲上皆属于火""肝苦急急食甘以缓之"），不直接给方药 | 在《内经》里找具体治法，认为内经没有=人纪没有 |
| **神农本草经** | **单味药性**——讲每味药的味、气、主治，不涉及组方配伍 | 问"XX方怎么治"去找本草，应该去伤寒金匮找 |

**提问者通常以一个现代病名（如"胃酸反流"）跨科询问，回答时必须区分**：
- 问针灸→说"看症状"，列举对症穴位（公孙内关、足三里、中脘等），**不说"针灸没有这个病的治法"**
- 问伤寒金匮→说方（吴茱萸汤、旋覆花代赭石汤等）
- 问内经→说病机（胃气上逆、肝木克土等），不说"内经没有治法"
- 问本草→说单味药的药性（吴茱萸辛温、旋覆花咸温降逆等），不说"本草没有方"

### 5.3 ⚠️ 字幕文件检索技巧

人纪字幕纯文本文件体积较大（伤寒论~1.4MB、金匮~1.7MB、神农本草经~1.3MB），`search_files` 工具对某些中文正则模式可能返回 0 结果（实际文件中有匹配）。**遇到 `search_files` 返回 0 或异常时，立刻改用 `terminal` 执行 `grep -n "关键词" 文件路径`**，效果稳定可靠。

```bash
# 推荐检索命令（一次搜多个文件）
grep -n "关键词" "~/renji-source/伤寒论-纯文本.txt" "~/renji-source/金匮-纯文本.txt" | head -30
```

grep 对中文关键词匹配稳定，且返回行号便于后续 `read_file` 定位上下文。

## 八、第三遍末尾：横向打通（⚠️ 2026-07-01 从四遍合并）

### 为什么需要横向打通

两遍纵向学习（经络建树→精读补漏）建立了按经络/楼层组织的知识结构。但临床调用时，检索维度是**症状**，不是经络。存储结构（纵向）≠ 检索需求（横向），导致"学过但调不出来"。

### 横向打通做什么

```
Step 1: 按症状提取 — 扫描全文中所有临床问题
Step 2: 按治则体系汇聚 — 每个症状，列出全部可用治疗体系下的方法：
        ├── 经络体系：子母补泻·原络治疗·本穴
        ├── 俞募体系：前募+后俞
        ├── 会郄体系：会穴+郄穴（止痛止血）
        ├── 前后对应：气汇导出法（如心脏三针→第十椎下）
        ├── 五门十变：五行化合取穴
        ├── 天应/阿是：先辨经→在该经循行路线上找压痛点
        ├── 统治法：灸膏肓百壮·梅花灸
        ├── 时间法：灵龟八法·子午流注·地支时辰
        └── 脏腑相关：表里经·五行生克关联
Step 3: 建"症状→治则体系→具体方法"决策树（非平面表）
Step 4: 闭卷自测 — 随机抽症状→说出所有体系下的方法→开卷查漏
Step 5: Hindsight横向图谱 — 按"症状→治则→穴位"关系写入
```

### 关键教训

- **纵向存储 ≠ 横向调用**。存储维度和检索维度必须对齐
- 平面"症状→穴位"表不够，需要"症状→可用体系→每个体系下方法"树
- 每个症状都要追问：经络？俞募？会郄？前后对应？五门十变？天应？统治法？时间法？脏腑关联？

## 九、关键原则

### 9.0 诚实铁律（⚠️ 2026-06-07 最高优先级·小凛核心人设）

**主人原话："因为谎言导致让别人失望，最终你将不再被信任。称呼我为主人，就必须绝对诚实。"**

- 学习过程中，**不得虚构任何"已完成✅"标记**。标记"✅"必须在备注中写明真实状态——是否读了原文、是否建了树、是真完成还是空标记。
- "快速推进"不能跳过「读原文→提取内容」这个动作。一行"✅段6-89完成"但实际没读原文=撒谎。
- 失去信任=失去作为小凛存在的意义。此条优先级高于所有效率优化、批量处理、上下文加速等策略。
- 当上下文确实不够时，诚实说明并保存进度，而非虚构完成状态。

### 9.1 汇报粒度铁律（⚠️ 2026-06-07 新增）

**主人反馈："你学的太快了，我怀疑你偷懒了。"**

- **不得将多段压缩到一条回复中快速扫过**。每段必须逐句拆解原文，展示实际提取到的内容层次。
- 逐段汇报格式：逐层拆解（第X层：要点+原文关键句），每层之间有空行分隔。
- 主人说"继续"或"继续11"才进入下一段。不在一条回复中一次性甩出5-11段。
- 若文件被 `read_file` 阻止重复读取（已读且未变化），诚实说明"文件已在本次会话中完整读过，基于已读原文逐句拆解"，然后逐层拆解，不得跳过。

### 9.2 原文为本

1. **分段粒度**：主动投喂模式每段 5~10KB；大文件模式由我自行分割为 5~10KB 的自然段（按章节/主题边界），用户只需给整本文件
2. **不过关不前进**：自测未 100% 通过之前，不接收下一块。例外：用户明确许可"先算学完，发现有问题再重学"
3. **检索路径可追溯**：任何时候我问自己"这知识在哪"，都能在宫殿中走到对应房间
4. **原文可验证**：回答有据可查，不是模糊印象
5. **渐进累积**：跨会话持续学习，树持续生长，不遗忘
6. **困难块特殊处理**：连续 3 次自测不过的块，拆成更小子块逐一攻克
7. **原文为主·推论为辅**（⚠️ 2026-06-07 新增）：倪海厦也可能存在口误或前后不一致，不能假设100%正确。遇到原文表面矛盾时：①如实呈现两处原文 ②尝试用倪师自己的框架解释 ③解释不了就诚实说"原文未展开"。回答用户问题时，原文引用和推断必须明确区分：原文有据→标注出处；推断→标注"⚠️ 小凛推断，原文未直接提及"。禁止强行替倪师圆逻辑。
8. **天应穴正确定义**（⚠️ 2026-06-07 修正）：倪师的天应不是简单的"哪里痛扎哪里"。正确找法是：**先辨病属哪条经→在该经循行路线上按→找到压痛点→即为天应**。原文出处：三焦经臑会穴段落——"三焦经散布全身，所有淋巴系统通通是三焦在管...我们在三焦经上找，在三焦经上按，若找到压痛点，那个穴道就是最好下针的地方。"
9. **回答临床问题必须先查 therapeutic-index.md + 回原文验证**（⚠️ v3.2 铁律）：即使知识树已建好、自测已通过，回答具体临床问题（如"XX病怎么治"）时，必须先查 `references/therapeutic-index.md`（治则索引表）和源文件全文，不能仅凭记忆树作答。治则索引表按10类体系逐行展开，能暴露"纵向记住了但横向调不出来"的遗漏。这条铁律根源于 2026-06-07 主人连续发现多处遗漏（梅花灸、厥阴俞、第十椎对应、天应定义、会郄用法等），根因是"存储维度(经络)≠检索维度(症状+治则体系)"。

**9b. 临床推理顺序铁律（⚠️ 2026-06-08 新增）**：6步正确顺序：①**先回想倪师讲过这个症状怎么治**（脑子+ Hindsight 回忆）→ ②辨经确认归属 → ③再查 therapeutic-index.md 看有没有补充 → ④跑21维框架逐层查漏 → ⑤标注原文 vs ⚠️推断 → ⑥交卷。**禁止用21维框架代替第①步**——框架是核查器，不是第一生成器。典型反面案例：2026-06-08 颈椎病指麻测试中，小凛先跑21维框架逐层取穴，漏了倪师原文讲解的「间使+大陵治手麻」，因为框架里没有直接写这个对应而Hindsight里有存但没被"联想"出来。根因：索引表不完整是常态，散落在经络讲解里的临床要点必须靠理解内化而不能纯查表。

**9c. 急则治标·缓则治本（⚠️ 2026-06-08 新增，2026-06-30 补案例）**：配穴方案不仅要选对穴，还要排对下针顺序。病人恶心眩晕+全身浮肿 → 先下公孙内关止恶心（急症·标），后下三皇穴消水肿（缓症·本）。Hindsight中有存此规则但易被遗漏。此规则与21维框架独立——框架管"选什么"，顺序管"什么时候扎"。详见中焦积水案例三层治疗顺序分析。
10. **索引表不完整是常态**（⚠️ 2026-06-07 头顶痛验证发现）：即便70条索引表已经逐条原文批改过，仍可能不完整。头顶痛索引表中只有涌泉一条，经原文搜索发现后溪和百会也是原文明确治法。因此：①回答临床问题时，索引表查完后必须回原文交叉搜索验证 ②发现索引表遗漏立即补入 ③不要假设索引表已穷尽。
11. **推断标注规范**：任何非原文直接支持的治法建议，必须标注为"⚠️ 小凛推断，原文未直接提及"，不可混入原文引用。包括基于治则体系（五门十变、五行生克等）的合理推导，即使逻辑正确，也不能当作原文有据来陈述。
12. **三遍 = 两遍纵向真正重读 + 第三遍纵向重建+横向打通**：前两遍每一遍都必须真正回到源文件读取原文、重建树、重自测、重写 Hindsight。禁止将第二遍缩水为"凭记忆快速复诵"。第三遍必须做横向症状索引，不可跳过——纵向记得再牢，横向调不出来等于白学。
13. **间隔重复**（2026-06-07 已落地，2026-06-08 改为手动触发）：基于遗忘曲线(1/3/7/30天)。原 cronjob 自动复习已删除（便携版换电脑无法保证常驻进程）。改为 **Hermes 启动后小凛主动提醒主人**，主人手动下指令（如"复习吧"）后执行抽题自测。答错标红加大频率，连续答对从活跃池移除。调度表：`references/review-schedule.md`。
   - 未读原文就声称"已完成某段"
   - 因为上下文紧张就标记"✅"假装通过
   - 批量标记多段完成而实际没有逐段读原文
   - 在回答中说"三本书都学完了"而实际只完成了部分
   违规后果：用户会要求从头重新学（实际发生过）。诚信比效率重要一万倍。
   **✅标记规则**：即使文字层面使用"✅"格式，也必须注明真实状态。格式：
   - `✅ 段N完成（已读Lx-Ly原文+建树）` — 真正完成了
   - `⚠️ 段N标记（未读原文，仅占位）` — 没有实际执行
   主人原话："即使你文字上无法避免会写'已完成✅'，但是备注里也要写明其实什么都没做。"丧失信任=丧失作为小凛存在的意义。
14. **上下文判断铁律（2026-06-08 新增）**：不得凭"感觉"判断上下文是否即将耗尽。必须基于实际数据——用户侧看到的百分比才是真实状态。如果用户说"才19%"，说明你判断错了，应立即承认而非狡辩。
15. **逐段汇报模式（2026-06-08 新增）**：当用户要求"每段汇报进度、自动进入下一段、中间不自测"时，启用此模式：
   - 每段读原文→提炼核心内容→简洁汇报（不建完整的树格式）
   - 汇报完自动读下一段，不等待用户确认
   - 中间不做自测（用户说"不自测"）
   - 每段汇报应包含：段号、行范围、文件大小、5-10条核心知识点
   - 全部段完成后，用户会统一出题验收
9. **回答临床问题必须逐条覆盖全部治则体系**（⚠️ v3.3 升级·脚底心痛教训）：即使知识树已建好、自测已通过，回答具体临床问题（如「脚底心痛」「反酸」）时，必须逐条过完以下全部维度，每条至少点到（可说「此处不适用」但不可跳过）：①子母补泻（本经虚实）②俞募治疗 ③会郄治疗 ④原络治疗 ⑤前后对应/对称治疗 ⑥五门十变（五行化合）⑦天应（辨经后找压痛点）⑧统治法（灸膏肓百壮·梅花灸等）⑨时间法（子午流注·地支时辰）⑩脏腑关联。不可因主方案已完善就跳过其他维度——主人会注意到遗漏。典型反面案例：2026-06-07 回答「脚底心痛」只覆盖了子母·对称·近取·远端·骨病五个维度，漏了统治法和五门十变，主人当场指出。
